# 一期范围整理

## 一期定位

一期目标不是做完整运维平台，而是先把“平台下发命令，Agent 稳定执行，状态可回看”这条闭环跑通。

## 一期已覆盖能力

### 平台侧

- Agent 接入、鉴权和在线状态维护
- 命令下发接口
- 命令执行状态跟踪
- 基础持久化与审计事件
- 第三方可调用的 HTTP API

### 执行侧

- 长连接与断线重连
- `update` / `restart` 两类基础动作
- 按目录串行，避免同目录 Compose 冲突
- 健康检查和运行状态暴露

### 工程侧

- 独立测试
- Docker 镜像构建
- Monorepo 统一 README、CI、Git 规则

## 一期建议的验收口径

详细执行步骤见 [`docs/PHASE1_ACCEPTANCE.md`](PHASE1_ACCEPTANCE.md)。

- Hub 能创建 Agent 并签发凭证
- Agent 能稳定连上 Hub 并定期保活
- Hub 能向指定 Agent 下发 `update` / `restart`
- Agent 能回传处理中和最终状态
- Hub 能保存最新状态和命令记录
- 两个服务都能独立通过基础测试和镜像构建

## 一期暂不纳入

- 多机器、多环境的复杂调度能力
- 面向很多角色的完整前端控制台
- 脱离 Docker Compose 的通用执行平台
- 很重的监控大盘、指标平台和报表体系
- 与当前远程运维闭环无关的流程型功能

## 一期整理后的仓库目标

这一轮仓库整理的目标，是让一期交付物更容易被继续开发，而不是大规模重写代码。重点是：

- 入口清晰
- 规则统一
- 文档成体系
- CI 能从 monorepo 根层工作
