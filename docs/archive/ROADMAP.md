# 后续路线规划

## 近阶段

- 统一镜像命名、环境变量命名和部署模板
- 把跨服务联调脚本接入 CI，减少手工回归
- 明确一期验收 checklist、故障排查步骤和发布流程
- 把命令失败结果整理成更稳定的结构，减少排障时依赖自由文本日志

## 下一阶段建议

- 梳理 Hub / Agent 间的消息契约，减少两边重复维护
- 补前置校验能力，例如目录、compose 文件、Docker 环境可用性检查
- 增加更清晰的执行结果结构，方便排查失败原因
- 完善 SQLite / MySQL 的部署说明、备份方式和迁移规范
- 视真实需要补一个最小可用页面，优先做“查看状态和命令记录”，不急着做复杂控制台

## 更后面的方向

- 如果后面确实出现明确场景，再考虑增加新的命令类型
- 如果 Docker Compose 不再满足需求，再评估是否支持其他执行方式
- 如果接入机器明显增多，再补更完整的观测、告警和运行统计
- 如果团队真的需要非研发同学也常用，再决定是否投入更完整的界面产品

## 组织建议

如果后面团队会继续扩人，推荐按照下面的节奏推进：

1. 先稳住一期闭环和交付质量。
2. 再减少协议重复和人工回归成本。
3. 最后只根据真实使用痛点扩展功能，不预设大而全的平台路线。

这样能避免系统还没稳定，就被一些听起来很“大”的概念带偏。
