插件分发改造方案

worker 零凭据 → 本机 agent →（持 token）平台；agent 兼做每主机插件缓存与拓扑自动发现，平台只维护「服务→插件→版本」。方案 A（agent 直连 platform）· 包走 HTTP 不走 WS。

service-agent / service-hub / service-platform cnp · docker/nocobase/sync-plugins.js 完整方案（非最小改造）

0设计原则

  1. worker 零凭据：worker 只跟本机 agent 说话，不持任何 token / URL 密钥。
  2. agent 一台一个，持本 namespace 的 pull-token（namespace.code = agentId 已是 1:1）。
  3. 分发链与控制链解耦，只在「发版投放」处汇合：发布新版本 → 控制链滚动重启 → 各实例经分发链重拉。
  4. 最大化复用现有契约：平台 /api/distribution/* 的 [{pluginName, version, url}] 正是为现有 sync-plugins.js 定的，字段不改名。
  5. 不发明新坑：插件包（.tgz）只走 HTTP，不过 agent↔hub 的 JSON/WS 通道。

2分发链详设（worker ↔ agent ↔ platform）

2.1 worker → agent 契约（新 · 本机 · 无 token）

扩展现有 core/health_server.py 的 ThreadingHTTPServer，或新起 PLUGIN_SERVE_PORT。

GET http://<agent-host>:<port>/plugins?service=<serviceCode>
  → 200 [{ "pluginName":"...", "version":"...", "url":"http://<agent-host>:<port>/download/<attachmentId>" }]
  （worker 无 namespace 概念——即便传了 agent 也忽略其值、恒用本 ns,故无法越 ns;url 指向 agent 自己,worker 无需 token）

  → 200 流式 .tgz（命中本机缓存直接给；未命中由 agent 回源平台后再给）
关键兼容：pluginName/version/url 三字段与平台端点字面一致 → sync-plugins.js 的解析与安装逻辑零改动。

2.2 agent → platform 回源（持 pull-token）

拉清单: GET {PLATFORM_URL}/api/distribution/plugins?namespace=<本ns>&service=<svc>
        Authorization: Bearer <agent pull-token>
改写:   把每条 url 改成指向「自己」/download/<attachmentId> 再回给 worker
下包:   /download/<id> 未命中缓存 → GET {PLATFORM_URL}/api/distribution/download/<id>
        Authorization: Bearer <pull-token> → 落缓存 → 回 worker

平台侧 完全复用现有端点（distribution.py）：query_plugins（token 属 namespace 校验 + 写 fetch_record）、download（IDOR 归属链 + 流式 .tgz）。

2.3 缓存（agent 的红利）

  • 内容寻址：键 = attachmentId，命中即用。
  • 一台只回源一次：同主机多容器同服务（admin/2admin）共享缓存。
  • 容量上限 + LRU；PLUGIN_CACHE_DIR 落盘，挂卷可跨重建保留。
  • 可选 sha256 校验（distribution 响应加字段，向后兼容）。
  • 并发同包 per-key 锁（借 handlers._get_project_lock 同款）。

2.4 鉴权边界

  • worker→agent 无 token：边界 = agent 的 namespace（1:1，天然成立）。
  • 同主机信任域；P2 可加「源 IP→容器→受管 compose 工程」校验。
  • agent→platform：pull-token namespace 隔离 + download IDOR（已实现）。

2.5 审计

  • agent 回源透传调用方维度 → 平台 fetch_record 记「哪个容器/服务拉了哪版本」。
  • agent 本地也记一份（平台不可达时仍有审计）。

3.4 服务纳管模型（意图 vs 现实）—— 服务不纯自动建

  • 物理节点（DiscoveredNode）= 纯自动：dir / 镜像 / 容器全来自 agent 发现，无手配。
  • 逻辑服务（Service，插件策略落点）= 不纯自动建。纯自动建有三硬伤：① 首启鸡生蛋（没配插件 → 拉空）；② 停了不能删（策略要留）→ 垃圾累积；③ 任意注册即建 → 管控弱。改走 自动发现名字 + 一键纳管 + 人工配插件。
  • 纳管：发现的 nacosService ∉ Service 表 → 进「已发现未纳管」收件箱；点「纳管」→ 平台预填（namespace + nacosServiceName，零敲）→ 建 Service → 配插件。保留手动预建给首次上线（敲一次 nacos 名绕开鸡生蛋）。Service 行持久，停了不自动删（策略要留）。
  • 对账视图（意图 Service ⋈ 现实 DiscoveredNode，by nacosServiceName）：running-but-unmanaged（在跑未纳管）· managed-but-down（纳管未起）· version-drift（版本漂移）。
  • link key（两列，不强制相等）：发现节点经 nacos_service_name link 到逻辑服务；分发链用 service_code（distribution.py 按它查清单）。二者是 db_models.py 上两个独立 nullable 列：创建表单默认把 service_code 预填成与 nacos_service_name 相同，但仍可分别编辑。删除"serviceCode = nacosServiceName 恒等"说法——nacos_service_name 现 P1 可空，恒等会让分发链拿到空 key。
结论：节点纯自动；服务这层「自动发现名字 + 一键纳管 + 人工配插件」，保留手动预建。不纯自动建服务（避免鸡生蛋 / 垃圾 / 管控弱）。

4控制链 / 投放（复用 node-control，不重造）

  • 发布新版本（改 service_plugin_version 的 active）→ 触发受影响 service 滚动重启（platform → hub rolling，逐实例 drain+restart）→ 各实例启动 → worker 经本机 agent 重拉（agent 已预热缓存，秒级）。
  • 版本一致性：平台 active 版本是单一真相；滚动逐实例，期间新起实例都拉同一目标版本。
  • 灰度（增强）：只滚一部分实例 / 按 DiscoveredNode 子集。
  • 预热：platform 通知 agent「新版本将至」→ agent 提前回源缓存 → 滚动时 worker 零等待。

4.1 ⚠️ 跨 agent（多节点）滚动的盲点 —— 必须 hub 协调

场景：两台服务器各一个 agent、共用一个 nacos，各跑一个同名服务 tamagawa-wms（该 nacos 服务 2 个实例，分属两台）。

当前实现的真实表现（守卫在 hub 编排器 rolling.py:_run_rolling，不在 agent）：rolling 命令定向单 agent、不广播；agent 的 handle_list_instances 逐实例只回 matched: true/false（从不失败），是 hub _run_rolling 见到非本机（unmatched）实例就整体 abort（测试 test_run_rolling_unmatched_aborts 直接驱动 hub）。所以给 agent A 发 rolling → hub 拿到含 B 的 unmatched 实例 → 整体 abort，滚不动。

结论：当前 rolling 只支持「同一台机多容器」（admin/2admin），不支持「一个服务跨多台机」——而后者正是真正的多节点生产形态。「两个 agent 同时重启 → 有中断」是另一种 naive 修法（去守卫各滚各的 / 广播并行）的后果。

正确做法：hub 跨 agent 协调的顺序滚动

  • hub 取该 nacos 服务全部实例，按 host→agent 分组（靠自动发现的「实例→agent」映射）。
  • 串行滚：agent A 的实例（drain→重启→等 ready）→ agent B → …，全程维持集群健康 ≥1（或 ≥quorum），绝不同时重启。
  • 「≥2 健康」门从「单机内」升级为「集群级」不变式；失败即停。
要改两处（均在 hub，非 agent）：① hub _run_rolling 把「有 unmatched 就 abort」→「过滤掉 unmatched、只滚本机 matched」（agent 无需改，它已逐实例回 matched 标志）；② hub 加跨 agent 顺序协调器（已能定向单 agent，补一层 for-each-agent 串行 + 集群健康门）。依赖自动发现（hub 须知每个实例在哪个 agent），与「agent 上报拓扑」配套。

4.2 服务的两个管理面 + 平台信息架构（与原型对齐）

服务（逻辑）挂两个独立管理、独立版本的维度，各有独立投放方式：

管理面管什么操作（服务二级页·编辑=暂存）发布机制（diff 自动选）
插件配置该服务装哪些插件 + 各插件 active 版本绑定 / 解绑、改版本（选版本）滚动重启 → worker 经 agent 重拉插件
镜像配置该服务的基础镜像（nocobase-pro tag）发布新镜像、历史、回滚pull-redeploy 逐节点
  • 编辑 = 暂存意图，发布 = 应用 diff（desired-state apply）：两个面只改 Service（逻辑层）的「意图」（暂存、标『待发布』）；一个「发布」按 diff 自动选机制——仅插件变 → 滚动 restart；含镜像变 → pull-redeploy（逐节点 pull/down/up，新容器启动顺带重拉插件，故插件变更自动包含）。不设两个并列发布（镜像发布 ⊃ 插件发布）。发布弹窗展示「变更摘要 + 机制」。
  • 创建服务只填 命名空间 + 服务编码（= nacos 服务名）；目录 / 镜像不在创建表单——目录纯节点级（自动发现），镜像走「镜像配置」管理面。
  • 镜像有历史 + 回滚：每次发布镜像入历史，可一键回滚到旧 tag（pull-redeploy）。

信息架构（侧栏）：配置（命名空间 / 服务 / 插件）· 发布（插件上传〔列表带「发布」=选 ns+服务〕/ 获取记录〔按 命名空间+服务+实例 聚合,插件@版本进详情〕；原「插件发布」独立页已删，发布统一走服务的「发布」）· 运维（节点 / 操作审计）。服务的「配置」「镜像」是服务二级页（从服务行进入，服务只读锁定，不在侧栏）；服务列表分 已纳管 / 未纳管 标签页（§3.4）；「实例」页（原「节点」，即 DiscoveredNode 的 UI 名）全自动发现（§3）。术语统一：命名空间 / 服务名（UI 去 (agent)/(nacos) 标注；底层 service_code 与 nacos_service_name 仍是两列、默认同值，见 §3.4）。实例运维动作简化为「启动 / 停止 / 更新」：更新 = 把该实例拉到服务期望态（按 diff 自动选 restart / pull-redeploy），取代分立的「重启 / 重部署」。

4.3 实例实时日志（agent 流式支持 —— 新）

实例页要能看实时日志（docker logs -f）。这是控制链上一个新的"流式"能力，与现有「一次性命令→结果」不同，要 agent 专门支持。

浏览器 ⇄ platform(SSE) ⇄ hub(WS) ⇄ agent → docker logs --tail 500 -f <container>
  • 为什么"要 agent 支持"：现 agent↔hub 是一次性命令→结果；实时日志是长连流式（持续吐行直到停）。新增 docker_cli.stream_logs(container, tail, follow)（subprocess 跑 docker logs -f 逐行 yield）+ 流式命令通道三类消息带 streamId：log-stream-start / log-chunk / log-stream-stop。
  • hub 转发 log-chunk 给订阅的 platform；platform 开 SSE 端点 /api/instances/{id}/logs/stream（admin 鉴权）经 hub 拉 agent 流推给浏览器。
  • 生命周期/防爆：默认 --tail 500 -f；仅实时不落库；关面板/断连/无活动超时 → 逐级 stop → agent kill docker logs 进程，不留孤儿；行速率截流 + 前端只留最近 ~500 行。
  • 权限（按 §5 取舍，内网从简）：走 admin-token→hub→agent 既有受信链；agent 只对本机受管容器开 logs；仅平台有权角色可看，不暴露 worker/public。
  • 数据模型：无新增持久表（纯流式）；归 P3，现状 ❌ 待建（流式通道是新能力，全链路要建）。

6健壮性（失败与启动）

  • worker 启动依赖本机 agent：agent 未起/无缓存且回源失败 → worker 回退上次已装插件（last-good，现有 sync-plugins.js 已具备），绝不白屏。
  • 启动顺序：agent 常驻 restart:always；worker 重启时 agent 已在。同 compose 可 depends_on。
  • agent 回源失败（平台挂/网络）→ 用缓存（可能旧版）兜底，标 stale；恢复后下次刷新。
  • 缓存损坏/校验失败 → 丢弃重拉；并发同包 per-key 锁。

  • platform：fetch_records（回源粒度，caller 取 agent 反查值；per-container 看 agent 本地日志，M2/M9）、DiscoveredNode 台账（在线/心跳/stale）、发布运行记录、发布→投放关联视图。
  • hub：命令审计（已有）。
  • 实例实时日志（§4.3）：实例页一键看 docker logs -f 实时 tail（浏览器 SSE ⇄ hub ⇄ agent 流式），仅实时不落库；离线/stale 实例不可用。

  "apiPath":  "/plugins",
  "namespace": "<ns>",          // 必填：现 mode2 以 namespace 非空为触发前提(L3)；agent 忽略其值、恒用本 ns
  "service":  "wms-test-admin"
  // 去掉 adminToken —— 本机 agent 无需 token
}

agent 端硬约束（对齐现有客户端，保证 worker 仅改配置）：/plugins?service= 返回纯数组 [{pluginName,version,url}] 字段不改名，url 指向 agent 自己。

⚠️ namespace 不是"可选可不传"(评审 L3 + G2)：现 mode2 进入条件含 && NAMESPACE，namespace 空会整段跳过同步(连 last-good 都不走)。二选一：(推荐·零代码改) 配置保留 namespace 字段(agent 忽略其值)；或要彻底去 ns 则必须改 mode2 gate(去 && NAMESPACE)——属必改非微调。
📌 跨仓 + 契约固化(评审 G1)：sync-plugins.js 在 cnp、本设计在 services-monorepo(两仓)；§9 的"已具备"都是对 cnp 当前实现的断言,漂移则 P2 真机静默失败。P0 把 worker↔agent 契约固化成 contract test(重定向/version 优先/tokenless/last-good/空清单不清空),cnp 脚本 pin commit + 负责人,改契约两仓同 PR。

11风险与权衡

  • agent 进 worker 启动关键路径 → last-good 缓存 + 启动顺序兜住；agent restart:always 常驻。
  • 同主机 tokenless 信任 → 接受（同信任域）；P2 可加容器来源校验。
  • 包不过 WS → 已定：列表走任意通道，包一律 HTTP。
  • nacos 只见在跑的 → docker（含 stopped）为主、nacos 补。
